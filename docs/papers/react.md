# ReAct：推理和行动交替

ReAct 的贡献不在复杂工程，而在把语言推理和外部行动组织成同一条可迭代轨迹。它让模型不只是一次性回答，也不只是盲目调用工具，而是在每轮行动后根据环境反馈修正下一步。

## 原文抓手

摘要里的关键句是：ReAct 让模型以 “interleaved manner” 生成推理轨迹和任务动作。

这句话的重点不是格式，而是依赖关系：推理帮助模型跟踪计划和处理异常；行动让模型从外部环境拿回新信息。也就是说，Thought、Action、Observation 不是三个标签，而是一种循环结构。

## 我的理解

ReAct 解决的是两个极端的问题：

| 模式 | 问题 |
| --- | --- |
| 只做 CoT | 推理可以很流畅，但事实可能来自模型内部猜测。 |
| 只做 Action | 能接触环境，但缺少显式计划、状态跟踪和综合判断。 |

ReAct 把二者放在一起：Thought 负责当前假设和下一步计划，Action 接触外部环境，Observation 把新事实放回上下文。

## 为什么现在看起来简单

今天看 ReAct 容易觉得“这不就是工具调用吗”。但在论文语境里，它重要的地方是把推理轨迹和环境交互统一成一条任务轨迹。Thought 不只是给人看的解释，它会进入上下文，影响下一步动作。

这也是它和普通工具调用的区别：工具调用可以是一次性的 API 使用，而 ReAct 关心多轮行动中的状态更新。

## 失败模式

ReAct 不是万能的。它仍然依赖：

- 搜索或环境返回的信息质量。
- 模型能否从 Observation 中正确更新计划。
- Prompt 格式是否足够稳定。
- 动作空间是否适合模型使用。

最后一点正好接到 SWE-agent：当 Action 变成“操作真实代码仓库”时，普通 shell 的反馈和动作空间就不一定适合语言模型。
